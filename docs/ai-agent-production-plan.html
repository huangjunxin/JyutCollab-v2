JyutCollab AI 助手生產級 Agent 改造計劃

本計劃取代早期正則路由式 workflow，把 Web AI 助手改造成 LLM 驅動的工具調用 Agent：LLM 負責理解意圖與規劃工具，代碼層負責工具白名單、Zod 校驗、權限、確認、安全審計與正式產品 UI。現況更新：Web Agent MVP 已具備 LLM runner、SSE 串流、只讀工具、本地導航工具、確認卡與基於 EditHistory 的寫入審計註記；下一個重點是把前端暫存聊天升級為可查詢、可審計、可開新對話的持久化聊天記錄系統。

Scope：Web AI Assistant 現況：Agent MVP 已落地 下一步：聊天歷史 / Conversation Store / AI Audit 不包含：Telegram / QQ / 飛書 安全原則：理解交給 LLM，安全交給代碼 目標：可上線且可審計

一、核心問題

正則路由不是 Agent

原有 includes('搜尋') 分支把自然語言理解鎖死在硬編碼 workflow 裡，導致「找一下粵拼為 ang1 的詞」這類明確需求也可能失敗。

工具 schema 不夠結構化

jyutcollab.search_entries 只有泛 query，缺少 jyutping、headword、category、register 等精確查詢欄位。

UI 仍像調試介面

Tool result 直接顯示 JSON，不符合上線產品體驗；應轉換為詞條列表、確認卡、導航卡等正式 UI。

安全邊界要與規劃分離

LLM 只規劃 tool call；工具白名單、schema validation、risk gate、actor-bound confirmation、審計由服務端代碼強制執行。

二、目前實作進度（2026-05-27）

已完成：Agent Core MVP

server/utils/agent/core/runner.ts 已使用 OpenAI-compatible chat completion + tool schemas，由模型根據自然語言選擇工具；工具名稱、輸入與輸出通過 registry 和 Zod 管理。

已完成：只讀與本地 UI 工具

readOnlyJyutCollab.ts 提供方言列表、詞條搜尋、詳情、查重；localUi.ts 提供本地導航，均不直接改寫資料庫。

已完成：Web 串流體驗

server/api/agent/chat.stream.post.ts 以 SSE 回傳狀態、文字片段、tool call delta、tool result；useAgentChat.ts 在側欄即時更新訊息、進度和工具卡。

已完成：確認與審計基礎

chat.post.ts 對草稿、送審、通過、拒絕使用 actor-bound confirmation + echo；成功寫入透過 EditHistory 記錄「via JyutCollab AI Agent」。

待強化：聊天記錄

目前訊息存放在前端 useState('agent-chat-messages')，刷新或換裝置後不可追溯；也沒有 conversations.get / [id].get 之類持久化 API。

待強化：AI Audit

現有 EditHistory 只覆蓋最終資料變更，尚未完整保存 conversation、tool call、模型回覆、確認請求、確認決策與失敗/阻擋事件。

三、目標架構

User message
  → Agent Runner (server/utils/agent/core/runner.ts)
  → LLM tool planning with registered tool schemas
  → Tool Registry + Zod validation
  → Risk / confirmation gate
  → Tool execution
  → Tool result fed back to LLM
  → Assistant response + productized UI payload

Agent loop 採用單線程循環：模型返回 tool call 就執行工具並把結果回流；模型返回純文本就結束。循環設置最大步數，防止失控。

五、聊天記錄與 AI 審計設計重點

能力建議資料產品行為審計要求
歷史對話列表AgentConversation：ownerId、title、channel、lastMessageAt、archivedAt、messageCount、lastSummary。側欄頂部提供「新對話」和歷史列表；只顯示當前登入用戶自己的對話。查詢必須按 ownerId 過濾；管理員審計入口另設權限，不混入普通列表。
訊息持久化AgentMessage：conversationId、role、content、createdAt、requestId、toolCall、confirmation、localAction。刷新後恢復當前 conversation；可切換舊對話繼續問。保存前走 redaction，不保存 raw request、headers、cookie、token。
新對話POST /api/agent/conversations 建立空 conversation；首次 user message 可自動生成 title。用戶點「新對話」後清空當前上下文，但不刪歷史。新 conversation 繼承 actor/channel，不繼承未完成 confirmation。
工具調用審計AgentAuditEvent：requestId、conversationId、actorId、eventType、toolName、risk、inputRedacted、outputSummary、blockedReason。普通用戶可看到工具卡；維護者可在審計頁查看事件鏈。每次 safe/local_ui/draft/editorial 工具都記錄；失敗和被阻擋也要記錄。
確認審計confirmationRequested / confirmationAccepted / confirmationRejected / confirmationExpired。確認卡展示 action、target、echo、後果和過期時間。必須記錄 actorId、target、risk、echo 是否匹配；不把用戶普通文字當確認。

六、資料庫集合建議

現有集合已包含 users、entries、edit_histories、ai_suggestions、notifications 等產品資料。聊天歷史和 AI 審計不應混入 edit_histories：edit_histories 繼續記錄詞條資料變更；Agent 互動本身用獨立集合保存，避免把「對話過程」和「詞條版本歷史」耦合。

集合用途核心欄位索引 / 保留策略
agent_conversations每個用戶的一條聊天會話；支援歷史列表、新對話、封存。ownerId、channel、title、lastMessageAt、messageCount、archivedAt、metadata.route、createdAt、updatedAt。索引：{ ownerId: 1, lastMessageAt: -1 }、{ ownerId: 1, archivedAt: 1 }。普通用戶查詢必須帶 ownerId。
agent_messages持久化 user / assistant / tool-visible messages，刷新後可恢復聊天。conversationId、ownerId、requestId、role、content、toolCall、confirmation、localAction、createdAt。索引：{ conversationId: 1, createdAt: 1 }、{ ownerId: 1, createdAt: -1 }。保存前 redaction；不存 raw headers/cookies/tokens。
agent_audit_events完整 AI audit trail：模型請求、工具調用、工具結果、確認請求/決策、阻擋、錯誤。conversationId、messageId、requestId、actorId、eventType、toolName、risk、inputRedacted、outputSummary、blockedReason、createdAt。索引：{ conversationId: 1, createdAt: 1 }、{ actorId: 1, createdAt: -1 }、{ requestId: 1 }。可按合規需求設 TTL；初期建議不自動刪除，先明確產品保留期。
edit_histories（沿用）只記錄詞條資料實際變更，例如 AI 建草稿、送審、通過、拒絕。沿用現有 entryId、userId、beforeSnapshot、afterSnapshot、changedFields、action、comment。在 comment 或新增 metadata 中關聯 agentConversationId/agentRequestId，但不要把完整聊天內容塞入 edit history。

最小可行方案是先加三個新集合：agent_conversations、agent_messages、agent_audit_events。如果之後確認要跨請求持久化未完成確認，再單獨加 agent_pending_confirmations；目前 chat.post.ts 的 in-memory pending actions 可先保留到下一輪強化。

七、安全與權限策略

  • safe：搜索、詳情、方言、查重，可自動執行。
  • draft_write / editorial：建立草稿、送審、通過、拒絕，必須生成 actor-bound confirmation，exact echo 通過後才執行。
  • destructive / admin：本階段不提供工具；後續必須額外授權和可審計日誌。
  • LLM 不接收 raw token、headers、cookie 或未清洗 payload；工具輸出統一通過 serializer redaction。

八、MVP 邊界

Web AI 助手可上線 MVP 已覆蓋自然語言查詞、結構化查詢、查重、詳情、方言、串流回覆、工具卡展示，以及草稿/送審/審核確認流程的安全保留。下一輪 MVP 邊界應集中在聊天記錄持久化與 AI audit trail：歷史對話列表、讀取舊對話、新增新對話、conversation-scoped tool/audit events、用戶只可查看自己的歷史。暫不實現 Telegram、QQ、飛書、群聊安全模型、跨實例 durable confirmation 和長期記憶壓縮。
