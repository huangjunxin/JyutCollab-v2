JyutCollab-v2 AI Agent Phase Plan

A concrete implementation plan for making AI assistance a reusable JyutCollab capability across Web sidebar, Telegram, Feishu, QQ, and MCP adapters while keeping permissions, confirmations, chat history, and audit behavior centralized. Current Web implementation has shipped the first Agent Core MVP; the next priority is durable conversation management so users can open new chats, review their own historical chats, and support AI audit trails.

Project: JyutCollab-v2 Stack: Nuxt 4 · Vue 3 · Nuxt UI · Nitro · Zod Primary UI: Web Sidebar Future Channels: Telegram · Feishu · QQ · MCP

0. Current Progress and Next Priority

Implemented in JyutCollab-v2

  • server/utils/agent/core/runner.ts runs an LLM-driven tool loop with registered Zod schemas.
  • server/utils/agent/tools/readOnlyJyutCollab.ts supports dialect list, search, detail, and duplicate checks with redacted payloads.
  • server/utils/agent/tools/localUi.ts supports safe Web navigation actions.
  • server/api/agent/chat.stream.post.ts streams status, text deltas, tool-call deltas, and tool results over SSE.
  • app/composables/useAgentChat.ts maintains sidebar state and renders streaming progress, tool cards, local actions, and confirmations.
  • server/api/agent/chat.post.ts gates draft, submit, approve, and reject actions with actor-bound confirmation and exact echo.
  • Successful AI-assisted writes currently leave EditHistory records such as “via JyutCollab AI Agent”.

Next priority: conversation history and audit

Current chat messages live in frontend state and are not durable. To support AI audit and user-visible history, the next slice should add persistent conversations, persisted messages, and structured audit events.

  • User can create a new conversation from the sidebar.
  • User can list and reopen their own historical conversations.
  • Every assistant response, tool call, confirmation request, decision, blocked action, and model/tool error has a conversation-scoped audit event.
  • Audit payloads must use the existing redaction policy and never store tokens, cookies, raw headers, or unfiltered request bodies.
Updated recommendation: before adding Telegram, Feishu, QQ, or MCP adapters to v2, finish Web conversation persistence and AI audit foundations so every later channel inherits durable history and traceability.

1. Architecture Principles

One core, many channels

Do not implement separate business tools for Web, Telegram, Feishu, QQ, and MCP. Build one Agent Core and expose it through channel adapters.

Agent Core
├── tools
├── permissions
├── confirmations
├── audit
└── LLM/tool loop

Channels
├── Web Sidebar
├── Telegram
├── Feishu
├── QQ
└── MCP

Never let AI bypass product rules

Agent tools must call existing domain logic or server utilities. They must not directly bypass validation, contribution permissions, review permissions, or confirmation gates.

  • Safe operations may run immediately.
  • Draft writes require light confirmation.
  • Editorial, destructive, and admin operations require strong confirmation.
Recommended stance: Treat the AI Agent as a new product surface over existing JyutCollab workflows, not as a privileged backend automation script.

4. Core Contracts

Tool definition

import { z } from 'zod'

export type AgentChannel = 'web' | 'telegram' | 'feishu' | 'qq' | 'mcp'

export type ToolRisk =
  | 'safe'
  | 'local_ui'
  | 'draft_write'
  | 'editorial'
  | 'destructive'
  | 'admin'

export interface AgentToolDefinition<Input, Output> {
  name: string
  description: string
  risk: ToolRisk
  inputSchema: z.ZodType<Input>
  outputSchema?: z.ZodType<Output>
  execute(input: Input, ctx: AgentToolContext): Promise<AgentToolResult<Output>>
}

Tool result

export interface AgentToolResult<T> {
  ok: boolean
  data?: T
  summary: string
  warnings?: string[]
  nextAction?: string
  confirmationRequired?: ConfirmationRequest
}

Confirmation request

export interface ConfirmationRequest {
  id: string
  action: string
  risk: ToolRisk
  actorId: string
  channel: AgentChannel
  target: {
    entryId?: string
    userId?: string
    headword?: string
    label: string
  }
  requiredEcho?: string
  expiresAt: string
  summary: string
  consequences: string[]
}

Channel capabilities

export interface ChannelCapabilities {
  canPatchFrontendState: boolean
  canRenderCards: boolean
  canUseButtons: boolean
  canSendPrivateReply: boolean
  canAccessCurrentPageContext: boolean
}

7. Global Acceptance Criteria

  • Agent Core owns tool policy, confirmation, permissions, conversation persistence, and audit behavior.
  • Users can create new conversations and reopen their own historical conversations.
  • AI audit events connect user message, assistant response, tool call/result, confirmation request/decision, and blocked/error outcomes by conversation id and request id.
  • Web sidebar, Telegram, Feishu, QQ, and MCP adapters do not duplicate JyutCollab business rules.
  • No write operation can be triggered by model text alone.
  • All editorial/destructive/admin operations require strong confirmation and server-side target verification.
  • Group confirmations are bound to the initiating actor.
  • Tool outputs never include auth headers, cookies, session tokens, raw request objects, or debug secrets.
  • All initial tools have tests for success, validation failure, permission failure, and confirmation failure.

8. Risks and Guardrails

RiskImpactGuardrail
Tool sprawl Model calls wrong or overly broad tools. Enable tools by page/channel capability; phase rollout; keep descriptions narrow.
Permission bypass AI performs operation user cannot do manually. Server-side permission checks in every write/editorial/admin tool.
Confirmation spoofing User text or another group member triggers mutation. Server-side confirmation id, actor id, target id, expiry, and echo validation.
Channel divergence Web and bot behaviors drift. One Agent Core; channel adapters only render/parse platform messages.
Missing audit continuity AI actions cannot be reconstructed after incidents or user disputes. Persist conversation messages and structured audit events for every tool call, confirmation, blocked action, and error.
History privacy leak User can view another user's AI conversation or audit metadata. Scope all conversation/message reads by owner id; keep admin audit access separate and explicitly permissioned.
Secret leakage Tokens or raw headers enter model context or chat output. Redaction helpers, token-safety tests, no raw payload returns.
Over-automation Users lose visibility into changes. Tool cards, previews, audit logs, and confirmation cards with consequences.
Non-goal for early phases: Do not implement admin/destructive automation before read-only, UI actions, draft writes, and review confirmations are stable and tested.
