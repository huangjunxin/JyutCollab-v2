<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User rule input -> client evaluator | Untrusted formula, regex, names, flags, and field choices cross into local evaluation. |
| Overlay metadata -> editable table cells | Derived warning/highlight state crosses into the existing inline-editing UI. |
| Overlay code -> existing entry objects | Read-only rule logic can read display contexts but must not mutate dictionary entries or dirty state. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01-01 | E | `useEntriesRuleOverlays.ts` formula handling | mitigate | Reuse `parseAdvancedFormula`/`evaluateAdvancedFormula`; static scan forbids `eval(`, `new Function`, and dynamic `import(`. |
| T-02-01-02 | D | `useEntriesRuleOverlays.ts` regex handling | mitigate | Reuse `compileAdvancedRegex`/`testAdvancedRegex`; no direct `new RegExp`; Phase 1 caps pattern/input/flags and rejects unsafe patterns. |
| T-02-01-03 | T | Entry object boundary | mitigate | Metadata maps are keyed by `getEntryIdString(entry)` and never assigned onto `entry`; grep forbids `_isDirty`, `col.set`, and entry underscored overlay state. |
| T-02-01-04 | R | Rule action semantics | mitigate | Expose only local add/toggle/remove/move/clear actions; no `$fetch`, save, delete, review, selection, or bulk action imports. |
| T-02-01-05 | I | Rule/error logging | accept | Composable should not log formula/regex input; rule inputs stay client-local and no secrets are involved. |
</threat_model>
