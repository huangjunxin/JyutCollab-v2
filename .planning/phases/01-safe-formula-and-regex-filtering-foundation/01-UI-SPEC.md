## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, validation icon/text gap, compact badge gap |
| sm | 8px | Control gaps inside advanced filter rows, button groups, field chips |
| md | 16px | Default panel spacing, table/header gaps, alert padding equivalent |
| lg | 24px | Advanced panel internal section gap when stacked |
| xl | 32px | Major page/content separation only; do not add inside table cells |
| 2xl | 48px | Empty-state icon/text block spacing and large vertical breathing room |
| 3xl | 64px | Page-level spacing only; not needed for this phase |

Exceptions:
- Preserve existing entries page classes where already established: search/filter shell uses `mb-4 p-3 gap-3 rounded-xl`; table cells use `px-3 py-2`; compact controls use `size="sm"` or `size="xs"`.
- Minimum interactive target: icon-only buttons must remain at least 32px visual size on desktop through Nuxt UI `size="xs"`/`size="sm"`; when shown on touch/mobile, controls must be reachable in a stacked panel with at least 40px height.
- Advanced field-name chips may use 6px horizontal padding only if implemented through existing Nuxt UI badge sizing; otherwise use 8px.

## Layout Contract

### Placement

Add advanced filtering inside the existing search/filter card in `app/pages/entries/index.vue`, directly under the current first row of search, server filters, view mode, and column-width actions.

Required structure:
1. Keep the existing search input and existing server-backed filters visually first.
2. Add a second row or disclosure panel inside the same `mb-4 p-3 bg-white dark:bg-gray-800 rounded-xl shadow-sm border` container.
3. The advanced row begins with a compact `進階篩選` disclosure/action area, followed by formula and regex controls when expanded.
4. The panel must not push the table controls outside the viewport on desktop; it may wrap to multiple lines using `flex flex-col xl:flex-row gap-3`.
5. The advanced panel must visually read as an extension of filtering, not as entry editing, rule management, conditional formatting, validation-rule management, or shareable-view UI.

### Density

Use the current table-filter density:
- Inputs: Nuxt UI `size="sm"`.
- Buttons: `size="sm"` for text buttons, `size="xs"` only for icon-only helpers.
- Panel border: use `border-t border-gray-200 dark:border-gray-700 pt-3 mt-3` if added inside the existing card.
- Avoid adding a modal or sidebar for Phase 1.

### Visual Hierarchy

1. Existing plain search remains dominant and unchanged.
2. Advanced filtering is secondary and opt-in.
3. Active advanced filters are summarized with one compact badge or text indicator next to `進階篩選`.
4. Validation errors sit directly below the offending input and above the table.
5. No-match empty state replaces the table only when the advanced filters are valid and active.

## Control Contract

### Formula Filtering

| Control | Component | Required Behavior |
|---------|-----------|-------------------|
| Formula input | `UInput` or `UTextarea` if one line proves too cramped | Accept both `=...` and formula text without `=`, but examples must show `=`. |
| Apply button | `UButton color="primary" variant="solid" size="sm"` | Applies current valid formula and regex state to currently loaded results. Disabled only while formula/regex validation is synchronously unresolved. |
| Clear button | `UButton color="neutral" variant="soft" size="sm"` | Clears formula text, global regex, column regex field, validation errors, and active advanced-filter state without changing existing `searchQuery` or server filters. |
| Helper tooltip or inline helper | `UTooltip`, `UAlert variant="subtle"`, or text block | Must list supported fields and functions near formula input. |

Formula field references are exactly: `headword`, `dialect`, `phonetic`, `entryType`, `theme`, `definition`, `register`, `status`.

Supported functions are exactly: `AND`, `OR`, `NOT`, `LEN`, `ISBLANK`, `REGEXMATCH`, `CONTAINS`, `STARTSWITH`, `ENDSWITH`.

### Regex Filtering

| Control | Component | Required Behavior |
|---------|-----------|-------------------|
| Global regex enable | Toggle-style `UButton` or checkbox-like control using Nuxt UI | Explicitly opts into regex mode. It must not change the existing global `searchQuery` behavior. |
| Global regex input | `UInput size="sm"` | Filters across concatenated supported display fields for currently loaded rows only. Empty value is inactive. |
| Column selector | `USelectMenu size="sm"` | Items are the supported public fields with human-readable labels. |
| Column regex input | `UInput size="sm"` | Filters only the selected column’s normalized display value. Empty value is inactive. |
| Flags control | Optional compact input/select | If exposed in Phase 1, allow only `i`, `m`, `u`; otherwise default to case-insensitive `i` internally. |

### Result Indicator

When any valid advanced filter is active, show the exact count helper: `進階篩選顯示 {visible} / {loaded} 項目前已載入結果。`

`visible` is the advanced-filtered visible entries count on the current loaded data. `loaded` is the count before the advanced layer, not `pagination.total`.

## State Contract

### Inactive State

When formula text is blank, global regex is disabled or blank, and column regex is blank:
- `tableRows`, `currentPageEntries`, `displayGroups`, empty-state logic, selection, AI suggestions, inline editing, dirty state, pagination, grouping, and keyboard navigation behave identically to the current entries table.
- Do not show the active count helper.
- Do not alter `useEntriesList` query params, cache key, or `/api/entries` requests.

### Valid Active State

When at least one advanced filter is valid and active:
- Filter the currently loaded `entries`, `aggregatedGroups`, or `lexemeGroups` client-side.
- Preserve original `Entry` object references so editing and dirty-state tracking still work.
- In aggregated and lexeme views, include group headers only when at least one child entry matches.
- Expanded groups display only matched child entries.
- The existing server-backed pagination controls remain based on server pagination; do not mutate `pagination.total`.
- Show the active count helper above the table or inside the advanced panel footer.

### Invalid Formula or Regex State

When a formula or regex pattern is invalid:
- Fail closed for that advanced rule only.
- Do not apply the invalid advanced filter.
- Do not clear existing table data.
- Do not refetch data.
- Do not mutate `searchQuery`, `filters`, `sortBy`, `sortOrder`, `currentPage`, or `pagination`.
- Keep the table showing the last valid advanced-filtered result if one exists; if no previous valid advanced filter exists, show the unadvanced current table.
- Display validation feedback inline near the relevant input using red/error styling.

### No-Match State

When all active advanced filters are valid but match zero rows:
- Use the existing empty-state pattern: centered panel, 80px circular neutral icon container, `i-heroicons-table-cells`, 18px semibold heading, 14px neutral body, optional clear button.
- Heading: `進階篩選沒有匹配結果`.
- Body: `沒有詞條符合目前公式或正則條件。請調整條件，或清除進階篩選返回原本結果。`
- Show `清除進階篩選` as the only CTA if advanced filters caused the empty state.
- Do not show `新建詞條` or `創建第一個詞條` in this advanced-filter no-match state.

## Interaction Contract

### Keyboard

- Existing table keyboard navigation remains unchanged when focus is inside the table.
- Advanced filter inputs follow normal form input keyboard behavior; arrow keys and text keys must not be intercepted by table keyboard handlers while focus is inside advanced controls.
- `Enter` inside formula/global regex/column regex input applies advanced filters.
- `Escape` inside advanced filter input clears only validation feedback focus/returns focus to the disclosure button if a popover/disclosure is used; it must not cancel table cell edits unless focus is in the table.
- `Tab` follows DOM order: existing search → server filters → view/width controls → advanced disclosure → formula → regex controls → apply → clear → table.

### Debounce and Responsiveness

- Validate formula/regex on input with a debounce of 200–300ms.
- Apply on explicit `套用進階篩選`; if live preview is implemented, it must still preserve the apply button and update within 300ms for typical loaded pages.
- Never block typing in existing table cells while advanced filters are active.
- If regex is rejected for length or unsafe pattern, show validation immediately and skip evaluation.

### Focus and Error Recovery

- On failed apply, focus remains in the invalid input.
- Error text must be announced via `role="alert"` or `aria-live="polite"`.
- `清除進階篩選` returns focus to the `進階篩選` disclosure/button and restores unadvanced table results.

## Accessibility Contract

- Every new input has a visible label or `aria-label` that matches the visible copy.
- Validation messages are programmatically associated with the input using `aria-describedby` when feasible.
- Error containers use `role="alert"` for failed apply and `aria-live="polite"` for debounced validation hints.
- The active advanced-filter summary is text, not color-only.
- Focus rings use existing Nuxt UI/Tailwind primary focus styling; do not remove outlines.
- Field-name and function helper content must be accessible without hover only; if using `UTooltip`, duplicate the essential list inline or in an expandable help text.
- Do not rely on red/green alone to communicate valid/invalid state.

## Responsive Behavior

### Desktop (`lg` and above)

- Existing search/filter row remains a single row where space allows.
- Advanced filter panel may use two rows:
  1. Formula input plus apply/clear.
  2. Regex controls plus helper text.
- Keep table height maximized; advanced panel should be compact and must not occupy more than roughly 160px when expanded at `lg` width.

### Tablet (`sm` to `lg`)

- Stack advanced formula and regex controls vertically inside the search card.
- Inputs use full width.
- Apply and clear buttons appear in one row with wrapping allowed.

### Mobile (`< sm`)

- Preserve existing mobile warning behavior.
- Advanced filters remain usable for browsing but must not imply mobile editing support.
- Stack all controls vertically with full-width inputs and buttons.
- Do not add horizontal-only controls that require scrolling inside the filter panel.
- Existing table horizontal overflow remains unchanged.

## Implementation Constraints

### Must Preserve

- Existing plain search semantics and `searchQuery` behavior.
- Existing server-backed filters for user, dialect, theme, status, sorting, pagination, and view mode.
- Existing grouping modes: flat, aggregated, lexeme.
- Existing inline editing, save/submit/delete, local draft recovery, AI suggestions, keyboard navigation, and column resizing.
- Existing localStorage draft watch over source `entries`, `aggregatedGroups`, and `lexemeGroups`; do not switch it to filtered collections.

### Must Not Build in Phase 1

- Conditional formatting UI.
- Validation rule management UI.
- Shareable view links or URL serialization.
- Saved views.
- Bulk modification, preview, or mutation actions driven by formulas/regex.
- Full spreadsheet grid replacement.
- Backend regex query changes.

### Safety Constraints

- Formula evaluation must use an explicit parser/interpreter and whitelist.
- Do not use `eval`, `new Function`, dynamic imports, arbitrary JavaScript execution, or executable user-provided code.
- Regex compilation must happen in a shared helper with `try/catch` and typed errors.
- Regex flags are limited to `i`, `m`, and `u`; do not allow `g` or `y` unless matcher resets `lastIndex` before every test.
- Recommended limits: 200-character regex pattern cap and 2,000-character tested display-string cap.
- Advanced filtering utilities must not receive or call `EditableColumnDef.set`, save/delete APIs, or mutation callbacks.
